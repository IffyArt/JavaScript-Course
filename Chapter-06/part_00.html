APP 是什麼？

App 實際上其實是 application 的簡稱，也就是應用程式的意思，而因為一些不可考證的因素，app 目前也高度指代 mobile application 或 mobile app 這兩個字組，如果總體的概念其實就是指「行動端應用程式」的意思．
而目前所謂 APP 的開發，在這幾年的技術發展下也產生了許多分支，由一開始的原生 APP 為主導開始，漸漸產生了 Hybrid(混合)、同時也人為了方便而設計了純 WebApp，而近幾年隨著技術的不斷堆疊與革新，人們也漸漸的對一些技術進行了更多的探索，進而產出了如 React Native 以及 Xamarin 等寫一套多平台享用的架構(自動編譯成所需平台)，以及將如 Java (Android)編譯為 Objective-C (iOS) 或相反已達到程式碼共用等技術，而如今我們主要將使用的操作，就是當中最為親民的 WEBAPP．


Native(原生語言)

基本介紹

原生的 APP 一般來說，泛指那些如 Android、iOS、Windows Phone 等作業系統，原先官方平台所使用來建立應用的語言，如目前 Android 非常大宗的 JAVA 或是近幾年興起的 Kotlin(官方正在推)，iOS 過往所使用的 ObjectC 以及目前主打的 Swift 等語言，所開發出的 APP 我們都稱為原生 APP (最符合作業系統原生架構)．


優點

  • 執行速度與效能均為最快
  • 社群龐大，官方也均提供了完整的教學與文件
  • 對於原先學習 Java 與 ObjectC 的開發者學習成本較低
  • 兩大作業系統(Android、iOS)相對穩定，技術練更換不會過快

缺點

  • 兩大平台使用語言不一，大多需要複數開發者且無法互相協作
  • 平台所提供之原生組件顯示效果不一(UI/UX設計師需顧慮差異)
  • 部分操作部分不同(UI/UX設計師需顧慮差異)


Hybrid( Native + Web )

基本介紹

以總結而言，就是在 iOS 與 Android 均有的 Webview 組件下手，再講簡單一點，就是在我們原生的基礎中，放置一個小的瀏覽器(已經封裝好直接用即可)，然後讓部分畫面可以介接外部網頁，如 Line 與 Facebook 等再開啟連結時，他們均會先以應用程式內的這個 Webview 為主，也因此 Hybrid 現階段作為主流的架構正被採用(儘管我認為僅僅部分連結使用 webview 就進行歸類很不合理)．

另外也有開發者會全以 Webview 為主，但這也代表了 Native 所提供的多種操作都將被捨棄． 封包網站


優點

  • 在大部分的情況下，保有了原生的全部功能以及操作
  • 混合式的開發模式，甚至能將部份 UI 操作交付給網頁應用程式
  • 接近於原生的特點，有益於進行問題的修復

缺點

  • 在應用程式內掛載瀏覽器的情況下，若未經優化效能總不讓人滿意(同時也要考慮優化成本是否過高)


WebApp

基本介紹

基本上其實就是一般的網頁應用程式，當中甚至不需要 RWD (儘管沒有將很難操作)，他完全作用於使用者行動裝置端上的瀏覽器，因此並不需要支付任何額外的成本，但相對的作業系統等平台原生的功能將很難使用(PWA 與 HTML5 有解決部份問題)


優點

  • 僅需要瀏覽器這個母體環境，較不受限於任何平台
  • 使用 RWD 的方式建置完成後，大部分支援全數平台
  • 不需要雙平台作業系統上傳 APP 的額外設定以及開發者費用
  • 操作見面以及流程較少因作業系統而導致差異
  • 部分原生功能持續開放給使用者中

缺點

  • 網站效能與原生應用無可比擬
  • 部分硬體操作等無法使用(藍芽等)
  • 較無法建立獨立的應用程式供使用者操作
目前有許多革新正在完善應用，但支援度並不理想

CrossApp (跨平台框架)

跨平台開發可說是如今業界的主流，因為實際上並沒有任何案子不需要雙平台的(除非特別討厭單一平台)，因此大部分我們都需要去處理雙平台的開發與維護，而當中絕大多數工程師並無法個人勝任如此重負，因此人力成本最少也會新增為兩人，而這就將帶來許多意料之外的問題與麻煩．

跨平台框架的建立，其實就是為了解決這種問題，他可以在一定程度上使用接近相同的 UI 進行開發，並且自動幫你編譯成改平台所適用的程式，當中也不乏有著封裝遊戲引擎等功能．

可以做到的主要代表

  • Unity3D - 目前非常大宗的遊戲引擎封裝，大家的遊戲開發入門好選擇
  • ReactNative - Facebook 想要把 React 做成只想學他就可以完成全部 Front End 的產物，若將 React 當作前端開發的主要工具，那他會是你的不二之選(雖然我們覺得他的實驗性質比實用價值高)
  • Ionic - Google 在 Angular 的基礎上開發的應用，跟 RN 概念類似，但本質其實是優化封裝起來的 Webview (至少在 4 版以前)
  • Titanium js - 與 RN 其實概念相似，但社群小「非常」多，玩味性質較高
  • Flutter - Google 使用 Dart 語言的全新力作，甚至在操作 UI 上幾乎不用考慮平台問題(有著完整的文件以及教學，還有著相對標準化的架構，若是純新手推薦學他)
  • Xamaris - Microsoft 推出的跨平台框架，因為使用 C# 所以門口其實沒有降低多少，且聽說還需理解原本作業系統的框架(概念上有點像撰寫共用的 Code)
  • AdobeFlash - 儘管他目前已經更新了一個新的應用程式，但使用者寥寥無幾
  • PhoneGap/Cordova - 目前已被 Adobe 收購，但本質其實就是 Webview

優點

  • 大部分情況只需要維護一套 Code
  • 有時部分技術練的學習成本會非常低
  • 不需考慮過多原生問題(嚴格說起來是考慮也沒用)

缺點

  • 根據使用的框架轉譯後的效能大多比不上原生應用
  • Debug 時很難界定問題是框架問題還是原生問題
  • 部份時候當功能無法滿足時，需花費大量的成本去呼叫原生應用
